# 工作守则

> 本文档定义了健身教练APP开发过程中的核心原则和工作流程，确保系统稳定性和开发效率

---

## 🎯 核心目标

**最终目标：** 开发一款健身教练和会员可以使用的APP

**核心要求：**
- ✅ 好用
- ✅ 稳定
- ⚠️ 不计成本，但稳定性优先于一切

---

## 📋 工作原则

### 原则 1：功能更新必须存档

**要求：**
- 每次完成一个功能后，必须立即进行 git 提交备份
- 提交信息要清晰描述完成的功能
- 确保随时可以从旧版本恢复

**执行方式：**
```bash
git add -A
git commit -m "完成功能：[功能描述]"
```

**时机：**
- 完成一个完整功能后
- 进行重大修改前
- 遇到问题需要大规模试错前

---

### 原则 2：遇到大问题先计划，后行动

**触发条件（满足任一即触发）：**
1. 需要大面积试错
2. 问题涉及核心依赖或系统配置
3. 可能影响系统稳定性的操作
4. 不确定解决方案的问题
5. 需要修改多个关键文件

**工作流程：**

#### 第一步：停止操作，切换到 Plan 模式
- 立即停止当前的试错操作
- 不要继续执行可能破坏系统的命令

#### 第二步：问题分析
1. **明确问题现象**
   - 具体的错误信息
   - 问题出现的时机
   - 影响范围

2. **分析问题根源**
   - 是什么导致的问题
   - 涉及哪些系统组件
   - 有哪些潜在风险

3. **列出可能的解决方案**
   - 方案 A：描述 + 优缺点 + 风险评估
   - 方案 B：描述 + 优缺点 + 风险评估
   - 方案 C：描述 + 优缺点 + 风险评估

#### 第三步：与用户沟通
- 清晰说明问题情况
- 列出所有可行方案
- 说明推荐方案及理由
- 等待用户确认后再执行

#### 第四步：执行方案
- 按照确认的方案执行
- 每一步都要验证结果
- 遇到新问题立即停止，重新进入 Plan 模式

---

### 原则 3：稳定性优先

**决策标准：**
- 当功能需求与系统稳定性冲突时，优先保证稳定性
- 宁可功能简单，不可系统崩溃
- 新功能必须在不影响现有功能的前提下添加

**风险控制：**
1. **高风险操作清单：**
   - 修改核心依赖（package.json）
   - 删除 node_modules
   - 修改数据库结构
   - 修改路由配置
   - 修改构建配置

2. **高风险操作前必须：**
   - 先进行 git 提交备份
   - 切换到 Plan 模式与用户沟通
   - 准备回滚方案

---

## 🔄 标准工作流程

### 日常功能开发流程

```
1. 接收需求
   ↓
2. 评估风险（低风险 → 直接开发 | 高风险 → Plan 模式）
   ↓
3. 开发功能
   ↓
4. 测试验证
   ↓
5. Git 提交备份
   ↓
6. 通知用户完成
```

### 问题处理流程

```
1. 发现问题
   ↓
2. 评估影响范围
   ↓
3. 判断是否需要 Plan 模式
   ├─ 需要 → 进入 Plan 模式 → 分析 → 沟通 → 执行
   └─ 不需要 → 直接修复 → 验证 → 提交
```

---

## ⚠️ 禁止事项

**绝对禁止在没有备份的情况下：**
- ❌ 删除 node_modules
- ❌ 修改 package.json 的核心依赖
- ❌ 执行数据库迁移
- ❌ 修改 .env 配置文件
- ❌ 大规模重构代码

**绝对禁止在没有沟通的情况下：**
- ❌ 尝试多个方案都失败后继续试错
- ❌ 遇到不熟悉的错误时盲目操作
- ❌ 修改系统级配置（Node 版本、npm 配置等）

## 📝 本次教训总结

**问题：** 安装 echarts 导致 vite 等核心依赖被移除，系统无法启动

**根本原因：**
- Node v25.6.1 版本过新，npm 存在兼容性问题
- 没有在修改依赖前进行备份
- 遇到问题后持续试错，没有及时切换到 Plan 模式

**应该怎么做：**
1. 安装新依赖前先 git 提交备份
2. 发现核心依赖被移除后，立即停止操作
3. 切换到 Plan 模式，分析问题根源
4. 列出解决方案（降级 Node、使用 pnpm、从备份恢复等）
5. 与用户沟通确认方案后再执行
