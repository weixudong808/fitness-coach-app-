## 🎯 核心业务逻辑

### 统一数据库 + 邀请码架构

**设计原则：**
- 所有教练和会员共用一个数据库
- 通过关系表管理教练-会员关系
- 邀请码机制：教练生成，会员使用，自动建立关系
- 主动选择机制：会员选教练，需要审核

**数据隔离：**
- 教练只能看到自己的会员数据
- 会员可以跟多个教练
- 教练可以带多个会员
- 训练计划永久保留（即使关系结束）

**取消关注逻辑：**
- 会员发起取消申请
- 教练1小时内处理：同意/拒绝
- 教练拒绝或1小时未处理 → 会员可强制取消
- 关系结束后，训练数据仍保留

### 6. RLS权限问题解决 ✅

**遇到的问题：**
- 教练/会员注册时报错："new row violates row-level security policy"
- 原因：前端使用 `VITE_SUPABASE_ANON_KEY`，匿名用户没有 INSERT 权限

**解决方案：**
创建 `fix-all-rls.sql` 脚本，包含：
- 允许匿名用户注册教练（INSERT 权限）
- 允许匿名用户注册会员（INSERT 权限）
- 允许任何人查看教练/会员（SELECT 权限，用于登录验证）
- 允许用户更新自己的数据（UPDATE 权限）

**执行方式：**
在 Supabase SQL Editor 中执行 `fix-all-rls.sql`
