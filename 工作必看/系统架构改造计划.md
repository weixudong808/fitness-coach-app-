## 改造背景

### 现状
- 每个教练通过 `coach_id` 隔离数据
- 使用 RLS（行级安全策略）控制权限
- 会员只能跟一个教练
- 没有教练注册和审核机制

### 问题
- 会员无法同时跟多个教练
- 教练无法在多个健身房工作
- 没有会员-教练匹配机制
- 数据归属不清晰

### 目标
- ✅ 统一数据库，所有教练和会员共用
- ✅ 会员可以跟多个教练
- ✅ 教练可以带多个会员
- ✅ 邀请码机制（教练生成，会员使用）
- ✅ 教练注册和审核
- ✅ 会员-教练关系管理
- ✅ 数据权限隔离（教练只能看自己的会员）

## 新架构设计

### 角色关系图

```
系统管理员（东哥）
    ↓ 审核
教练A、教练B、教练C...
    ↓ 邀请码/主动添加
会员1、会员2、会员3...
    ↓ 可以跟多个教练
训练计划（按教练分类）
```

### 核心逻辑

**1. 教练注册流程**
```
教练填写注册信息
→ 提交申请
→ 管理员审核
→ 审核通过 → 教练可以登录
→ 审核拒绝 → 显示拒绝原因
```

**2. 会员注册流程**
```
会员填写注册信息（含初始体重）
→ 注册成功
→ 提示：要不要选择教练？
   ├─ 选择：进入教练列表，选教练，发送申请
   └─ 跳过：进入会员主页，浏览"我的认证"等页面
```

**3. 会员-教练匹配（两种方式）**

**方式A：邀请码（推荐）**
```
教练生成邀请码
→ 教练把邀请码发给会员（微信、短信等）
→ 会员注册时输入邀请码
→ 自动建立关系（不需要审核）
```

**方式B：主动选择**
```
会员在"教练页面"选择教练
→ 发送申请
→ 教练同意/拒绝（拒绝需填写原因）
→ 同意后建立关系
```

**4. 取消关注流程**
```
会员点击"取消关注教练"
→ 发送取消申请
→ 教练1小时内处理：
   ├─ 同意：关系结束
   └─ 拒绝：出现"强制取消"按钮
→ 教练1小时未处理：自动出现"强制取消"按钮
→ 会员点击"强制取消"：关系强制结束
→ 训练计划等数据永久保留
```

## 数据库改造

### 1. 新增表

#### ① coaches（教练表）
```sql
CREATE TABLE coaches (
  id UUID PRIMARY KEY DEFAULT uuid_generate_v4(),
  name TEXT NOT NULL,                    -- 教练姓名
  phone TEXT UNIQUE NOT NULL,            -- 手机号（唯一）
  password TEXT NOT NULL,                -- 密码（加密存储）
  audit_status TEXT DEFAULT 'pending',   -- 审核状态：pending/approved/rejected
  reject_reason TEXT,                    -- 拒绝原因
  created_at TIMESTAMP DEFAULT NOW(),
  updated_at TIMESTAMP DEFAULT NOW()
);

-- 索引
CREATE INDEX idx_coaches_phone ON coaches(phone);
CREATE INDEX idx_coaches_audit_status ON coaches(audit_status);
```

#### ② invite_codes（邀请码表）
```sql
CREATE TABLE invite_codes (
  id UUID PRIMARY KEY DEFAULT uuid_generate_v4(),
  coach_id UUID NOT NULL,                -- 教练ID
  code TEXT UNIQUE NOT NULL,             -- 邀请码（唯一）
  is_active BOOLEAN DEFAULT true,        -- 是否有效
  created_at TIMESTAMP DEFAULT NOW(),
  FOREIGN KEY (coach_id) REFERENCES coaches(id) ON DELETE CASCADE
);

-- 索引
CREATE INDEX idx_invite_codes_code ON invite_codes(code);
CREATE INDEX idx_invite_codes_coach_id ON invite_codes(coach_id);
```

#### ③ coach_member_relations（教练-会员关系表）
```sql
CREATE TABLE coach_member_relations (
  id UUID PRIMARY KEY DEFAULT uuid_generate_v4(),
  coach_id UUID NOT NULL,                -- 教练ID
  member_id UUID NOT NULL,               -- 会员ID
  status TEXT DEFAULT 'pending',         -- 关系状态：pending/active/ended
  initiator TEXT NOT NULL,               -- 申请方：coach/member/invite_code
  reject_reason TEXT,                    -- 拒绝原因
  end_type TEXT,                         -- 结束类型：coach_delete/member_cancel/member_force_cancel
  start_time TIMESTAMP,                  -- 开始时间
  end_time TIMESTAMP,                    -- 结束时间
  cancel_request_time TIMESTAMP,         -- 取消申请时间（用于计算1小时超时）
  created_at TIMESTAMP DEFAULT NOW(),
  updated_at TIMESTAMP DEFAULT NOW(),
  FOREIGN KEY (coach_id) REFERENCES coaches(id) ON DELETE CASCADE,
  FOREIGN KEY (member_id) REFERENCES members(id) ON DELETE CASCADE,
  UNIQUE(coach_id, member_id, status)    -- 同一教练和会员只能有一个active关系
);

-- 索引
CREATE INDEX idx_relations_coach_id ON coach_member_relations(coach_id);
CREATE INDEX idx_relations_member_id ON coach_member_relations(member_id);
CREATE INDEX idx_relations_status ON coach_member_relations(status);
```

#### ④ notifications（通知表）
```sql
CREATE TABLE notifications (
  id UUID PRIMARY KEY DEFAULT uuid_generate_v4(),
  user_type TEXT NOT NULL,               -- 用户类型：coach/member/admin
  user_id UUID NOT NULL,                 -- 用户ID
  type TEXT NOT NULL,                    -- 通知类型：coach_audit/member_apply/coach_apply/cancel_request等
  title TEXT NOT NULL,                   -- 通知标题
  content TEXT NOT NULL,                 -- 通知内容
  is_read BOOLEAN DEFAULT false,         -- 是否已读
  related_id UUID,                       -- 关联ID（如关系ID）
  created_at TIMESTAMP DEFAULT NOW()
);

-- 索引
CREATE INDEX idx_notifications_user ON notifications(user_type, user_id);
CREATE INDEX idx_notifications_is_read ON notifications(is_read);
CREATE INDEX idx_notifications_created_at ON notifications(created_at DESC);
```

### 2. 修改现有表

#### ① members（会员表）- 添加字段
```sql
-- 添加初始体重字段
ALTER TABLE members ADD COLUMN initial_weight DECIMAL(5,2);  -- 初始体重（kg）
ALTER TABLE members ADD COLUMN current_weight DECIMAL(5,2);  -- 当前体重（kg）
ALTER TABLE members ADD COLUMN password TEXT;                -- 密码（加密存储）

-- 修改手机号为唯一
ALTER TABLE members ADD CONSTRAINT members_phone_unique UNIQUE(phone);

-- 添加索引
CREATE INDEX idx_members_phone ON members(phone);
```

**说明：**
- `initial_weight`：注册时填写，不能改，用于认证基准
- `current_weight`：会员可以更新，用于当前认证计算
- 高级体能认证用 `current_weight` 计算

#### ② training_templates（训练模板表）- 保持不变
```sql

-- 但需要注意：现在 coach_id 指向 coaches 表，不再是 auth.users
```

#### ③ member_plans（会员训练计划关联表）- 保持不变
```sql
-- 不需要修改，coach_id 表示"哪个教练给会员做的计划"
```

### 3. RLS 策略调整

#### 教练表 RLS
```sql
-- 启用 RLS
ALTER TABLE coaches ENABLE ROW LEVEL SECURITY;

-- 教练可以查看自己的信息
CREATE POLICY "教练查看自己的信息"
ON coaches FOR SELECT
USING (id = auth.uid());

-- 教练可以更新自己的信息
CREATE POLICY "教练更新自己的信息"
ON coaches FOR UPDATE
USING (id = auth.uid());
```

#### 关系表 RLS
```sql
-- 启用 RLS
ALTER TABLE coach_member_relations ENABLE ROW LEVEL SECURITY;

-- 教练可以查看自己的会员关系
CREATE POLICY "教练查看自己的会员关系"
ON coach_member_relations FOR SELECT
USING (coach_id = auth.uid());

-- 会员可以查看自己的教练关系
CREATE POLICY "会员查看自己的教练关系"
ON coach_member_relations FOR SELECT
USING (member_id = auth.uid());
```

#### 通知表 RLS
```sql
-- 启用 RLS
ALTER TABLE notifications ENABLE ROW LEVEL SECURITY;

-- 用户只能查看自己的通知
CREATE POLICY "用户查看自己的通知"
ON notifications FOR SELECT
USING (
  (user_type = 'coach' AND user_id = auth.uid()) OR
  (user_type = 'member' AND user_id = auth.uid())
);

-- 用户可以标记自己的通知为已读
CREATE POLICY "用户标记通知已读"
ON notifications FOR UPDATE
USING (
  (user_type = 'coach' AND user_id = auth.uid()) OR
  (user_type = 'member' AND user_id = auth.uid())
);
```

## 风险评估

### 高风险

#### 1. 数据迁移风险
**问题：** 现有测试数据可能与新结构不兼容

**解决方案：**
- 改造前完整备份数据库
- 删除所有测试数据，从零开始
- 不做数据迁移，直接用新结构

#### 2. RLS 策略配置错误
**问题：** RLS 配置错误可能导致数据泄露或无法访问

**解决方案：**
- 仔细测试每个 RLS 策略
- 使用不同账号测试权限隔离
- 记录所有 RLS 策略配置

### 中风险

#### 3. 邀请码冲突
**问题：** 邀请码可能重复

**解决方案：**
- 使用 UUID 或随机字符串生成邀请码
- 数据库设置 UNIQUE 约束
- 生成时检查是否已存在

#### 4. 1小时超时逻辑
**问题：** 取消关注1小时超时逻辑可能不准确

**解决方案：**
- 使用数据库时间戳计算
- 前端定时检查（每分钟）
- 后端 API 返回是否超时

### 低风险

#### 5. 通知系统性能
**问题：** 通知数量多时可能影响性能

**解决方案：**
- 添加索引优化查询
- 分页加载通知
- 定期清理已读通知
