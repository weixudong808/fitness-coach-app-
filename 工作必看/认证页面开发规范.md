## 1️⃣ 经验值进度条

### 数据来源
**来自：我的徽章墙**

### 计算规则
- 统计徽章墙中所有已点亮的徽章数量
- 每点亮一个徽章，经验值增加一定百分比
- 公式：`经验值百分比 = (已点亮徽章数 / 总徽章数) × 100%`

### 示例
- 总徽章数：20个
- 已点亮：5个
- 经验值：25%

### 实现要点
- ✅ 实时计算，不依赖缓存
- ✅ 徽章墙数据变化时，经验值自动更新
- ✅ 数据来源统一：徽章墙的点亮状态

## 2️⃣ 打卡认证

### 数据来源
**来自：`training_sessions` 表**

### 计算规则
统计会员所有训练计划中 `completed = true` 的课次数量

**统计范围：**
- ✅ 统计"进行中"状态的训练计划（`status = 'active'`）
- ✅ 统计"已完成"状态的训练计划（`status = 'completed'`）
- ❌ 不统计"已取消"状态的训练计划（`status = 'cancelled'`）
- ✅ 只统计 `completed = true` 的课次
- ❌ 不统计"待训练"的课次（`completed = false`）

### 认证等级
| 等级 | 打卡次数 | 徽章状态 |
|------|---------|---------|
| 1级 | 10次 | 已点亮/未点亮 |
| 2级 | 25次 | 已点亮/未点亮 |
| 3级 | 50次 | 已点亮/未点亮 |
| 4级 | 75次 | 已点亮/未点亮 |
| 5级 | 100次 | 已点亮/未点亮 |

### 徽章点亮规则
- 当前打卡次数 ≥ 目标次数 → 徽章点亮（金色）
- 当前打卡次数 < 目标次数 → 徽章未点亮（灰色）

### 实现要点
- ✅ 实时计算，不依赖 `member_achievements` 表
- ✅ 认证列表和徽章墙使用同一份数据
- ✅ 数据来源统一：`training_sessions` 表

### SQL 查询示例
```sql
-- 查询会员的打卡次数
SELECT COUNT(*) as check_in_count
FROM training_sessions
WHERE template_id IN (
  SELECT template_id FROM member_plans
  WHERE member_id = '会员ID'
  AND status IN ('active', 'completed')
)
AND completed = true;
```

## 4️⃣ 基础体能认证

### 状态
✅ **已确认**

### 数据来源
**来自：`session_exercises` 表（训练计划中的动作记录）**

### 性别区分显示
根据会员注册时的性别字段，显示对应的认证标准：

**男生认证标准（6项）：**
1. 俯卧撑 15次
2. 引体向上 1次
3. 靠墙静蹲 90秒
4. 平板支撑 2分钟
5. 波比跳 20次/2分钟
6. 坐姿体前屈

**女生认证标准（6项）：**
1. 跪姿俯卧撑 15次
2. 反向划船 25次
3. 靠墙静蹲 90秒
4. 平板支撑 90秒
5. 波比跳 20次/2分钟
6. 坐姿体前屈

### 计算规则

**自动同步逻辑：**
1. 扫描会员所有已完成的训练记录（`completed = true`）
2. 查找训练动作中是否包含基础体能考核项目
3. 检查动作的完成数据是否达到认证标准
4. 如果达标，自动标记该项认证为完成
5. 相应徽章自动点亮

**判断标准：**
- 动作名称必须完全匹配（如"跪姿俯卧撑"）
- 重量要求：自重（或符合标准）
- 次数/时长：达到或超过标准值
- 自动更新：无需手动操作

### 示例

**女生会员 - 跪姿俯卧撑认证：**
- 认证要求：跪姿俯卧撑 15次
- 训练记录：跪姿俯卧撑 自重 20次
- 判断：20次 ≥ 15次 → ✅ 认证完成
- 结果：该项徽章点亮

**男生会员 - 引体向上认证：**
- 认证要求：引体向上 1次
- 训练记录：引体向上 自重 3次
- 判断：3次 ≥ 1次 → ✅ 认证完成
- 结果：该项徽章点亮

### 数据流向关系

```
训练计划（training_sessions + session_exercises）
    ↓
基础体能认证进度计算（实时计算）
    ↓
基础体能徽章显示
```

**重要原则：**
- ✅ 基础体能徽章不独立计算数据
- ✅ 徽章直接使用"认证进度"中已经计算好的数据
- ✅ 保证数据一致性：认证进度显示的和徽章墙显示的是同一份数据

### 实现要点
- ✅ 根据会员性别动态显示对应的认证项目
- ✅ 实时扫描训练记录，自动判断是否达标
- ✅ 动作名称需要精确匹配
- ✅ 支持次数和时长两种类型的判断
- ✅ 6项全部完成后，基础体能认证徽章点亮
- ✅ 徽章墙使用认证进度的计算结果（共用数据）

### SQL 查询示例
```sql
-- 查询会员是否完成某项基础体能认证（以跪姿俯卧撑为例）
SELECT EXISTS (
  SELECT 1
  FROM session_exercises se
  JOIN training_sessions ts ON se.session_id = ts.id
  JOIN member_plans mp ON ts.template_id = mp.template_id
  WHERE mp.member_id = '会员ID'
  AND ts.completed = true
  AND se.exercise_name = '跪姿俯卧撑'
  AND se.weight IN ('自重', '')
  AND CAST(REGEXP_REPLACE(se.reps_standard, '[^0-9]', '', 'g') AS INTEGER) >= 15
) as is_completed;
```

根据会员注册时的性别字段，显示对应的认证标准：

1. **体脂率认证**：14-17%
2. **上肢力量认证**：

5. **心肺功能认证**：

**女生认证标准（5项）：**
1. **体脂率认证**：21-24%
2. **上肢力量认证**：
   - 卧推：体重×0.5，8次

   - 战绳：1.5分钟

### 计算规则

**自动同步逻辑：**
1. 获取会员初始体重（`members.weight`）
2. 扫描会员所有已完成的训练记录（`completed = true`）
3. 查找训练动作中是否包含高级体能考核项目
4. 提取动作的重量、次数、组数
5. 判断是否达到认证标准：
   - 重量：训练重量 ≥ 会员初始体重 × 倍数标准
   - 次数：训练次数 ≥ 要求次数
6. 如果达标，自动标记该项认证为完成
7. 相应徽章自动点亮

**判断标准：**
- 动作名称必须完全匹配（如"卧推"）
- 重量计算：基于会员注册时的初始体重（固定值）
- 次数/时长：达到或超过标准值
- 自动更新：无需手动操作

### 示例

**男生会员 - 卧推认证：**
- 会员初始体重：80kg
- 认证要求：卧推 80kg（体重×1.0）8次
- 训练记录：卧推 85kg 10次 3组
- 判断：
  - 重量：85kg ≥ 80kg ✅
  - 次数：10次 ≥ 8次 ✅

**女生会员 - 深蹲认证：**

  - 次数：8次 ≥ 8次 ✅
  - 结果：下肢力量认证（深蹲项）完成

### SQL 查询示例
```sql
-- 查询会员是否完成卧推认证（男生为例）
WITH member_info AS (
  SELECT weight FROM members WHERE id = '会员ID'
)
SELECT EXISTS (
  SELECT 1
  FROM session_exercises se
  JOIN training_sessions ts ON se.session_id = ts.id
  JOIN member_plans mp ON ts.template_id = mp.template_id
  WHERE mp.member_id = '会员ID'
  AND ts.completed = true
  AND se.exercise_name = '卧推'
  AND CAST(REGEXP_REPLACE(se.weight, '[^0-9.]', '', 'g') AS NUMERIC) >= (SELECT weight FROM member_info) * 1.0
  AND CAST(REGEXP_REPLACE(se.reps_standard, '[^0-9]', '', 'g') AS INTEGER) >= 8
) as is_completed;
```

## 6️⃣ 我的徽章墙

### 数据来源
**来自：认证进度的实时计算结果（共用数据）**

### 数据流向关系

```
训练计划（training_sessions + session_exercises）
    ↓
认证进度计算（实时计算）
    ↓
徽章墙显示
```

**重要原则：**
- ✅ 徽章墙不独立计算数据
- ✅ 徽章墙直接使用"认证进度"中已经计算好的数据
- ✅ 保证数据一致性：认证进度显示的和徽章墙显示的是同一份数据
- ✅ 避免重复查询数据库，提高性能

### 徽章分类
1. **新手会员徽章**
   - 数据来源：会员注册状态
   - 点亮规则：会员注册成功即点亮

2. **打卡认证徽章**（5个）
   - 数据来源：打卡认证的实时计算结果
   - 点亮规则：打卡次数达到目标即点亮

3. **影响力认证徽章**
   - 数据来源：待确认
   - 点亮规则：待确认

4. **基础体能认证徽章**
   - 数据来源：基础体能认证的实时计算结果
   - 点亮规则：6项全部完成后点亮

5. **高级体能认证徽章**
   - 数据来源：高级体能认证的实时计算结果
   - 点亮规则：5项全部完成后点亮

### 实现要点
- ✅ 徽章墙直接使用各认证模块的数据
- ✅ 不单独查询数据库
- ✅ 数据来源统一，避免不一致
- ✅ 只计算一次，多处使用（性能优化）

### 代码实现
```javascript
// 页面加载时，先计算各类别的认证进度
const loadAchievements = async () => {
  const memberId = await getCurrentMemberId()

  // 1. 计算各类别的认证进度（只计算一次）
  checkInAchievements.value = await getAchievementsByCategory(memberId, 'check_in')
  newbieAchievements.value = await getAchievementsByCategory(memberId, 'newbie')
  influenceAchievements.value = await getAchievementsByCategory(memberId, 'influence')
  basicFitnessAchievements.value = await getAchievementsByCategory(memberId, 'basic_fitness')
  advancedFitnessAchievements.value = await getAchievementsByCategory(memberId, 'advanced_fitness')

  // 2. 徽章墙直接使用这些已经计算好的数据（共用数据）
  allAchievements.value = [
    ...newbieAchievements.value,
    ...checkInAchievements.value,
    ...influenceAchievements.value,
    ...basicFitnessAchievements.value,
    ...advancedFitnessAchievements.value
  ]
}

// 判断徽章是否点亮
const isAchieved = (code) => {
  const achievement = allAchievements.value.find(a => a.code === code)
  return achievement?.progress?.is_completed === true
}
```

### 为什么选择"共用数据"方案？

**对比分析：**

| 方案 | 优点 | 缺点 |
|------|------|------|
| **方案1：共用数据**（✅ 采用） | 数据一致性好、性能高、代码简洁、易维护 | 无明显缺点 |
| **方案2：各算各的**（❌ 不采用） | 独立性强 | 数据可能不一致、性能差、代码冗余、难维护 |

**结论：** 采用"共用数据"方案，符合"单一数据源"原则，是最佳实践。

## 🔧 开发原则

### 1. 数据流向统一（核心原则）

**数据流向关系：**
```
训练计划（原始数据）
    ↓
认证进度计算（实时计算）
    ↓
徽章墙显示（共用数据）
```

**实施要点：**
- ✅ 训练计划是唯一的原始数据源
- ✅ 认证进度负责计算和处理数据
- ✅ 徽章墙直接使用认证进度的计算结果
- ✅ 同一个数据只计算一次，多处使用
- ❌ 禁止徽章墙独立查询数据库
- ❌ 禁止认证进度和徽章墙分别计算

### 2. 数据来源统一
- 同一个数据只能有一个来源
- 认证列表和徽章墙必须使用同一份数据
- 禁止分别查询导致数据不一致

### 3. 实时计算优先
- 优先使用实时计算，避免依赖缓存表
- 缓存表容易导致数据不同步
- 实时计算保证数据准确性

### 4. 避免重复查询
- 一次查询，多处使用
- 通过 composable 共享数据
- 减少数据库压力

### 5. 测试验证
- 每次修改后必须测试
- 验证认证列表和徽章墙数据一致性
- 测试各种边界情况（0次、达标、超标）
