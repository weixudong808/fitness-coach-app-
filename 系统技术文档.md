## 项目概述

### 项目背景
健身教练管理系统，用于管理会员信息、训练计划、训练记录等。支持多教练、多会员同时使用。

### 核心需求
1. **训练记录功能** - 教练通过对话方式实时记录会员训练数据
2. **文档暂存机制** - 先保存到 Markdown 文档，稍后批量导入数据库
3. **智能导入** - 自动查询会员信息、创建训练计划、处理重名问题
4. **多教练支持** - 每个教练独立配置，数据隔离

### 技术选型
- **前端：** Vue 3 + Element Plus
- **后端：** Supabase (PostgreSQL + RLS)
- **脚本：** Node.js + ES Modules
- **文档格式：** Markdown

## 技术架构

### 系统架构图
```
┌─────────────────┐
│   前端 (Vue 3)   │
│  Element Plus   │
└────────┬────────┘
         │ Anon Key (RLS 限制)
         ↓
┌─────────────────┐
│    Supabase     │
│   PostgreSQL    │
│   RLS Policies  │
└────────┬────────┘
         ↑
         │ Service Role Key (绕过 RLS)
┌────────┴────────┐
│  后台脚本 (Node) │
│  - 导入训练记录  │
│  - 初始化数据    │
│  - 数据验证      │
└─────────────────┘
```

### 权限模型
1. **前端查询** - 使用 Anon Key，受 RLS 策略限制，只能查询自己教练的数据
2. **后台脚本** - 使用 Service Role Key，绕过 RLS，可以操作所有数据
3. **教练隔离** - 通过 `coach_id` 字段实现多教练数据隔离

## 数据库结构

### 核心表结构

#### 1. members（会员表）
```sql
CREATE TABLE members (
  id UUID PRIMARY KEY DEFAULT uuid_generate_v4(),
  name TEXT NOT NULL,              -- 会员姓名
  phone TEXT,                       -- 手机号
  gender TEXT,                      -- 性别 (male/female)
  age INTEGER,                      -- 年龄
  created_at TIMESTAMP DEFAULT NOW()
);
```

#### 2. training_templates（训练模板表）
```sql
CREATE TABLE training_templates (
  id UUID PRIMARY KEY DEFAULT uuid_generate_v4(),
  name TEXT NOT NULL,               -- 模板名称
  description TEXT,                 -- 描述
  target_goal TEXT,                 -- 训练目标
  difficulty_level TEXT,            -- 难度等级 (beginner/intermediate/advanced)
  training_stage TEXT,              -- 训练阶段
  is_template BOOLEAN DEFAULT true, -- 是否为公共模板
  member_id UUID,                   -- 会员专属模板的会员ID
  coach_id UUID NOT NULL,           -- 教练ID（关键字段！）
  created_at TIMESTAMP DEFAULT NOW(),
  FOREIGN KEY (member_id) REFERENCES members(id)
);
```

**重要说明：**
- `is_template = true` - 公共模板，所有会员可用
- `is_template = false` + `member_id` - 会员专属模板（如"张三-日常训练记录"）
- `coach_id` - **必须设置为真实的教练ID**，否则前端无法查询到数据

#### 3. training_sessions（训练课次表）
```sql
CREATE TABLE training_sessions (
  id UUID PRIMARY KEY DEFAULT uuid_generate_v4(),
  template_id UUID NOT NULL,        -- 所属模板ID
  session_number INTEGER NOT NULL,  -- 课次编号
  core_focus TEXT,                  -- 核心重点
  training_part TEXT,               -- 训练部位
  completed BOOLEAN DEFAULT false,  -- 是否完成
  completed_date DATE,              -- 完成日期
  created_at TIMESTAMP DEFAULT NOW(),
  FOREIGN KEY (template_id) REFERENCES training_templates(id)
);
```

#### 4. session_exercises（课次动作表）
```sql
CREATE TABLE session_exercises (
  id UUID PRIMARY KEY DEFAULT uuid_generate_v4(),
  session_id UUID NOT NULL,         -- 所属课次ID
  exercise_name TEXT NOT NULL,      -- 动作名称
  weight TEXT,                      -- 重量 (如 "80kg")
  reps_standard TEXT,               -- 次数标准 (如 "10次")
  sets INTEGER,                     -- 组数
  coach_comment TEXT,               -- 教练评语
  order_index INTEGER,              -- 排序索引
  created_at TIMESTAMP DEFAULT NOW(),
  FOREIGN KEY (session_id) REFERENCES training_sessions(id)
);
```

#### 5. member_plans（会员训练计划关联表）
```sql
CREATE TABLE member_plans (
  id UUID PRIMARY KEY DEFAULT uuid_generate_v4(),
  member_id UUID NOT NULL,          -- 会员ID
  template_id UUID NOT NULL,        -- 模板ID
  coach_id UUID NOT NULL,           -- 教练ID（关键字段！）
  start_date DATE,                  -- 开始日期
  status TEXT DEFAULT 'active',     -- 状态 (active/completed/cancelled)
  created_at TIMESTAMP DEFAULT NOW(),
  FOREIGN KEY (member_id) REFERENCES members(id),
  FOREIGN KEY (template_id) REFERENCES training_templates(id)
);
```

### RLS 策略说明
前端使用 Anon Key 查询时，RLS 策略会检查 `coach_id` 是否匹配当前登录的教练。因此：
- ✅ `coach_id` 必须设置为真实的教练ID
- ❌ 不能使用 `member.id` 作为 `coach_id`（这是之前的bug）

## 核心功能

### 1. 训练记录功能

#### 工作流程
```
1. 教练对话 → "开启张三训练记录"
2. AI 创建文档 → 训练记录-张三-2026-03-05.md
3. 教练输入动作 → "卧推 80kg 10次 3组"
4. AI 记录到文档 → Markdown 表格格式
5. 教练说"结束" → 保存文档，状态：待导入
6. 运行导入脚本 → node import-training-record.js
7. 脚本处理：
   - 查询会员信息（支持重名识别）
   - 查询或创建"会员名-日常训练记录"模板
   - 确定下一个课次编号
   - 创建新课次（completed=false）
   - 批量插入动作数据
   - 验证前端可见性
   - 更新文档状态为"已导入"
```

#### 文档格式
```markdown
# 训练记录

**会员：** 张三
**训练部位：** （用户说了就填，没说就留空）
**训练日期：** 2026-03-05
**开始时间：** 19:30

---

## 训练动作

| 序号 | 动作名称 | 重量 | 次数 | 组数 | 备注 |
|------|---------|------|------|------|------|
| 1 | 卧推 | 80kg | 10次 | 3 | |
| 2 | 哑铃飞鸟 | 20kg | 12次 | 3 | |

---

**状态：** 待导入
**记录时间：** 2026-03-05 19:30 - 20:15
**总动作数：** 2
```

### 2. 智能导入功能

#### 重名处理
如果数据库中有多个同名会员，导入时会提示：
```
找到 2 个名为"张三"的会员：
1. 张三 (13800138000)
2. 张三 (13900139000)
请在文档中指定：**会员：** 张三 (8000)
```

#### 自动创建模板
如果会员没有"日常训练记录"模板，脚本会自动创建：
- 模板名称：`{会员名}-日常训练记录`
- `is_template = false`
- `member_id = 会员ID`
- `coach_id = 配置文件中的教练ID`

#### 课次编号管理
自动查询该模板下已有的最大课次编号，新课次编号 = 最大编号 + 1

## 核心脚本

### 1. import-training-record.js（训练记录导入）

**功能：**
- 扫描 `训练记录/` 目录，找到所有"待导入"状态的文档

- 查询或创建"会员名-日常训练记录"模板
- 创建新课次和动作数据
- 验证前端可见性（模拟 Anon Key 查询）
- 更新文档状态为"已导入"

**使用方法：**
```bash
node import-training-record.js
```

**关键代码：**
```javascript
// 使用配置的教练ID
const coachId = process.env.COACH_ID

// 创建模板时使用真实教练ID
const { data: newTemplate } = await supabase
  .from('training_templates')
  .insert({
    name: templateName,
    coach_id: coachId,  // ✅ 正确

// 导入后验证前端可见性
const { data: verifyPlans } = await supabaseAnon

  .eq('member_id', member.id)
```

## 问题解决记录

### 问题1：RLS 策略阻止脚本操作

**现象：**
```
Error: new row violates row-level security policy
```

**原因：**
脚本使用 Anon Key，受 RLS 策略限制，无法写入数据。

**解决方案：**
使用 Service Role Key 绕过 RLS：
```javascript
const supabaseServiceKey = process.env.SUPABASE_SERVICE_ROLE_KEY
const supabase = createClient(supabaseUrl, supabaseServiceKey)
```

---

### 问题2：数据库字段不匹配

**现象：**
```
Error: column "is_exclusive" does not exist
```

**原因：**
脚本使用的字段名与实际数据库不一致。

**解决方案：**
更新脚本，使用正确的字段名：
- ❌ `is_exclusive` → ✅ `is_template`
- ❌ `exclusive_member_id` → ✅ `member_id`
- ❌ `duration_weeks` → ✅ `difficulty_level`
- ❌ `sessions_per_week` → ✅ `training_stage`

---

### 问题3：导入成功但前端看不到数据（核心问题）

**现象：**
- 脚本显示"导入成功"
- 数据库中有数据
- 前端查询返回空数组

**原因：**
```javascript
// ❌ 错误代码
coach_id: member.id  // 使用了会员ID作为教练ID
```

前端使用 Anon Key 查询时，RLS 策略检查 `coach_id` 是否匹配当前登录的教练。由于 `coach_id` 被错误地设置为 `member.id`，导致 RLS 过滤掉了这些记录。

**解决方案：**
1. 在 `.env.local` 中配置真实的教练ID
2. 脚本从环境变量读取教练ID
3. 创建模板和训练计划时使用真实教练ID

```javascript
// ✅ 正确代码
const coachId = process.env.COACH_ID

const { data: newTemplate } = await supabase
  .from('training_templates')
  .insert({
    coach_id: coachId,  // 使用真实教练ID
    member_id: member.id
  })
```

4. 导入后自动验证前端可见性

```javascript
// 模拟前端查询
const supabaseAnon = createClient(supabaseUrl, supabaseAnonKey)
const { data: verifyPlans } = await supabaseAnon
  .from('member_plans')
  .select('*, training_templates (*)')
  .eq('member_id', member.id)

if (!verifyPlans || verifyPlans.length === 0) {
  console.error('⚠️ 警告：前端无法查询到导入的数据！')
}
```

---

### 问题4：训练部位自动推断可能出错

**现象：**
AI 根据动作名称自动推断训练部位，可能识别错误。

**解决方案：**
更新 CLAUDE.md 规则：
- ✅ 用户说了训练部位就记录
- ✅ 没说就留空，不自动推断
- ✅ 教练后续可以自己补充
