# 代码实施前检查清单

> 每次准备修改代码前，必须完整过一遍这个清单，确保逻辑正确、不出bug

---

## 📋 第一步：需求理解确认

### 1. 完全理解用户需求
- [ ] 我是否完全理解了用户想要什么？
- [ ] 是否有任何模糊或不确定的地方？
- [ ] 如果有疑问，是否已经和用户讨论清楚？

### 2. 反馈理解给用户
- [ ] 用自己的话总结用户的需求
- [ ] 列出关键点，让用户确认
- [ ] 等待用户确认"理解对了"再继续

**禁止行为：**
- ❌ 自己猜测用户需求
- ❌ 理解不清楚就开始写代码
- ❌ 跳过确认环节

---

## 🔍 第二步：方案讨论（如果有多种方案）

### 1. 列出所有可行方案
- [ ] 方案A：描述 + 优缺点
- [ ] 方案B：描述 + 优缺点
- [ ] 方案C：描述 + 优缺点

### 2. 从两个角度分析
- [ ] **IT技术角度：** 哪个方案实施更简单？哪个不容易出bug？
- [ ] **用户角度：** 哪个方案用户体验更好？更符合实际使用场景？

### 3. 给出建议并等待确认
- [ ] 明确推荐哪个方案
- [ ] 说明推荐理由
- [ ] 等待用户确认后再实施

**禁止行为：**
- ❌ 只有一个方案就直接做
- ❌ 不讨论就自己决定用哪个方案
- ❌ 只考虑技术角度，不考虑用户体验

---

## 🧪 第三步：逻辑验证（写代码前）

### 1. 先写逻辑测试代码
- [ ] 不要直接修改项目代码
- [ ] 先写一个独立的测试脚本
- [ ] 用模拟数据测试逻辑是否正确

### 2. 测试所有场景
- [ ] 正常场景：功能正常工作
- [ ] 边界场景：空值、null、undefined
- [ ] 异常场景：错误数据、未来日期、重复数据
- [ ] 极端场景：数据量很大、时间跨度很长

### 3. 确认逻辑100%正确
- [ ] 所有测试场景都通过
- [ ] 没有遗漏的情况
- [ ] 逻辑清晰，没有歧义

**禁止行为：**
- ❌ 直接修改代码，边写边测试
- ❌ 只测试正常场景，不测试边界情况
- ❌ 逻辑不确定就开始写代码

---

## 💻 第四步：代码实施

### 1. 找到正确的修改位置
- [ ] 阅读相关代码，理解现有逻辑
- [ ] 确定需要修改哪些文件
- [ ] 确定需要修改哪些函数

### 2. 最小化修改
- [ ] 只修改必要的部分
- [ ] 不要重构无关代码
- [ ] 保持代码风格一致

### 3. 添加必要的验证
- [ ] 空值检查
- [ ] 类型检查
- [ ] 边界条件检查
- [ ] 错误处理

**禁止行为：**
- ❌ 大面积修改代码
- ❌ 顺便重构其他代码
- ❌ 缺少错误处理

---

## 🎭 第五步：角色扮演测试

### 1. 把自己当成健身教练
- [ ] 我是教练，我会怎么使用这个功能？
- [ ] 按照真实工作流程操作
- [ ] 不是技术测试，而是用户体验测试

### 2. 测试所有使用场景
- [ ] 场景1：正常使用流程
- [ ] 场景2：用户可能犯的错误
- [ ] 场景3：特殊情况处理
- [ ] 场景4：边界情况

### 3. 记录所有问题
- [ ] 功能bug
- [ ] 操作不流畅
- [ ] 数据显示不对
- [ ] 逻辑不符合实际使用场景

### 4. 发现问题先反馈
- [ ] 不要急着改
- [ ] 先反馈给用户
- [ ] 讨论确认后再修改

**禁止行为：**
- ❌ 只做技术测试（点击按钮看有没有报错）
- ❌ 发现bug立即修改（可能理解错需求）
- ❌ 不测试就提交代码

---

## 📊 第六步：提交代码

### 1. Git提交前必须备份
- [ ] **重要：每次修改代码前先 git commit**
- [ ] 确保有一个干净的恢复点
- [ ] 提交信息要清晰，说明修改前的状态

### 2. 再次检查
- [ ] 代码逻辑正确
- [ ] 所有测试通过
- [ ] 没有破坏现有功能
- [ ] 代码格式正确

### 3. 写清晰的提交信息
- [ ] 说明解决了什么问题
- [ ] 说明如何解决的
- [ ] 列出测试场景
- [ ] 添加 Co-Authored-By

### 4. Git提交
- [ ] `git add -A`
- [ ] `git commit -m "详细的提交信息"`
- [ ] 确认提交成功

**禁止行为：**
- ❌ 提交信息不清楚
- ❌ 没有测试就提交
- ❌ 忘记提交
- ❌ 修改代码前不备份

---

## 🔙 第七步：出错时的撤回机制

### 核心原则：出现大面积错误，立即撤回，不要修修补补

**什么情况需要撤回：**
- 新功能导致大面积报错
- 系统无法正常运行
- 破坏了现有功能
- 出现连锁反应的bug

**正确做法：**
1. ✅ 立即停止修改
2. ✅ 使用 `git reset --hard HEAD~1` 撤回到上一个提交
3. ✅ 或者使用 `git revert` 撤销提交
4. ✅ 确保系统恢复到正常状态
5. ✅ 重新分析问题，找到正确的解决方案
6. ✅ 重新实施

**错误做法：**
- ❌ 在出错的基础上继续修改
- ❌ 试图修修补补解决问题
- ❌ 不撤回，一直往前走
- ❌ 让系统处于不稳定状态

**撤回命令：**
```bash
# 方法1：硬重置到上一个提交（丢弃所有修改）
git reset --hard HEAD~1

# 方法2：撤销提交但保留修改（可以重新修改）
git reset --soft HEAD~1

# 方法3：创建一个新的撤销提交（保留历史记录）
git revert HEAD
```

**为什么要撤回：**
- ✅ 保证系统始终处于可用状态
- ✅ 避免在错误的基础上越改越乱
- ✅ 重新思考，找到正确的解决方案
- ✅ 不浪费时间在错误的方向上

## ⚠️ 特别注意事项

### 1. 数据库相关
- [ ] 使用正确的日期字段（session_date vs completed_date）
- [ ] 只统计已完成的训练（completed=true）
- [ ] 考虑数据为空的情况
- [ ] 考虑数据格式不一致的情况

### 2. 日期相关
- [ ] 使用实际训练日期，不是标记完成的日期
- [ ] 不要限制时间范围（除非明确要求）
- [ ] 处理未来日期的情况
- [ ] 处理日期格式（YYYY-MM-DD vs ISO格式）

### 3. 用户体验
- [ ] 错误提示要清晰明确
- [ ] 不要让用户困惑
- [ ] 操作流程要符合常识
- [ ] 考虑用户可能犯的错误

### 4. 性能考虑
- [ ] 查询时使用索引
- [ ] 只加载需要的数据
- [ ] 考虑数据量大的情况
- [ ] 避免重复查询

---

## 🎯 核心原则

### 1. 先讨论，后实施
- 理解需求 → 反馈确认 → 讨论方案 → 确认方案 → 测试逻辑 → 实施代码

### 2. 逻辑优先，代码其次
- 先确保逻辑100%正确
- 再写代码实现
- 不要边写边想

### 3. 用户视角，不是技术视角
- 站在健身教练的角度思考
- 不是站在程序员的角度
- 功能要符合实际使用场景

### 4. 发现问题先反馈
- 不要自己猜测
- 不要急着改
- 先讨论，后修改

### 5. 保证一遍过
- 充分测试
- 逻辑正确
- 不出bug
- 用户满意

---

## ✅ 检查清单总结

**每次准备写代码前，问自己：**
1. ✅ 我是否完全理解了需求？
2. ✅ 用户是否确认了我的理解？
3. ✅ 如果有多个方案，是否讨论并确认了？
4. ✅ 逻辑是否已经测试验证？
5. ✅ 是否考虑了所有边界情况？
6. ✅ 是否站在用户角度思考了？
7. ✅ 是否准备好了角色扮演测试？

**如果有任何一项是"否"，停止写代码，先完成那一项！**
