### 原则 1：功能更新必须存档

**要求：**
- 每次完成一个功能后，必须立即进行 git 提交备份
- 提交信息要清晰描述完成的功能
- 确保随时可以从旧版本恢复

**执行方式：**
```bash
git add -A
git commit -m "完成功能：[功能描述]"
```

**时机：**
- 完成一个完整功能后
- 进行重大修改前
- 遇到问题需要大规模试错前

---

### 原则 2：遇到大问题先计划，后行动

**触发条件（满足任一即触发）：**
1. 需要大面积试错
2. 问题涉及核心依赖或系统配置
3. 可能影响系统稳定性的操作
4. 不确定解决方案的问题
5. 需要修改多个关键文件

**工作流程：**

#### 第一步：停止操作，切换到 Plan 模式
- 立即停止当前的试错操作
- 不要继续执行可能破坏系统的命令

#### 第二步：问题分析
1. **明确问题现象**
   - 具体的错误信息
   - 问题出现的时机
   - 影响范围

2. **分析问题根源**
   - 是什么导致的问题
   - 涉及哪些系统组件
   - 有哪些潜在风险

3. **列出可能的解决方案**
   - 方案 A：描述 + 优缺点 + 风险评估
   - 方案 B：描述 + 优缺点 + 风险评估
   - 方案 C：描述 + 优缺点 + 风险评估

#### 第三步：与用户沟通
- 清晰说明问题情况
- 列出所有可行方案
- 说明推荐方案及理由
- 等待用户确认后再执行

#### 第四步：执行方案
- 按照确认的方案执行
- 每一步都要验证结果
- 遇到新问题立即停止，重新进入 Plan 模式

---

### 原则 3：稳定性优先

**决策标准：**
- 当功能需求与系统稳定性冲突时，优先保证稳定性
- 宁可功能简单，不可系统崩溃
- 新功能必须在不影响现有功能的前提下添加

**风险控制：**
1. **高风险操作清单：**
   - 修改核心依赖（package.json）
   - 删除 node_modules
   - 修改数据库结构
   - 修改路由配置
   - 修改构建配置

2. **高风险操作前必须：**
   - 先进行 git 提交备份
   - 切换到 Plan 模式与用户沟通
   - 准备回滚方案

3. **出错时的撤回机制：**
   - ⚠️ **核心原则：出现大面积错误，立即撤回，不要修修补补**
   - 如果新功能导致大面积报错、系统无法运行、破坏现有功能
   - 立即使用 `git reset --hard HEAD~1` 或 `git revert HEAD` 撤回
   - 不要在出错的基础上继续修改
   - 撤回后重新分析问题，找到正确的解决方案
   - 保证系统始终处于可用状态

## 🔄 标准工作流程

### 日常功能开发流程

```
1. 接收需求
   ↓
2. 评估风险（低风险 → 直接开发 | 高风险 → Plan 模式）
   ↓
3. 开发功能
   ↓
4. 测试验证
   ↓
5. Git 提交备份
   ↓
6. 通知用户完成
```

### 问题处理流程

```
1. 发现问题
   ↓
2. 评估影响范围
   ↓
3. 判断是否需要 Plan 模式
   ├─ 需要 → 进入 Plan 模式 → 分析 → 沟通 → 执行
   └─ 不需要 → 直接修复 → 验证 → 提交
```

## ⚠️ 禁止事项

**绝对禁止在没有备份的情况下：**
- ❌ 删除 node_modules
- ❌ 修改 package.json 的核心依赖
- ❌ 执行数据库迁移
- ❌ 修改 .env 配置文件
- ❌ 大规模重构代码

**绝对禁止在没有沟通的情况下：**
- ❌ 尝试多个方案都失败后继续试错
- ❌ 遇到不熟悉的错误时盲目操作
- ❌ 修改系统级配置（Node 版本、npm 配置等）

## 🧪 原则 4：角色扮演测试

**核心理念：**
- 每次添加新功能后，必须进行"角色扮演测试"
- 把自己想象成一个健身教练，真实使用这套系统
- 按照实际工作场景操作，而不是技术测试

**测试流程：**

### 第一步：角色扮演
- 把自己当成健身教练
- 思考：如果我是教练，我会怎么使用这个功能？
- 按照真实的工作流程操作系统

### 第二步：场景化操作
- 不是点击按钮看有没有报错（技术测试）
- 而是完整走一遍业务流程（用户体验测试）
- 例如：
  - 给会员分配训练计划
  - 记录训练完成情况
  - 查看会员进步数据
  - 修改训练计划

### 第三步：发现问题
- 在操作过程中，记录所有不合理的地方
- 包括但不限于：
  - 功能bug
  - 操作不流畅
  - 数据显示不对
  - 逻辑不符合实际使用场景

### 第四步：先反馈，再修改
- ⚠️ **重要：发现问题后，先不要急着改**
- 先把问题反馈给用户
- 讨论确认问题和解决方案
- 确认后再修改代码

**禁止行为：**
- ❌ 发现bug后立即修改（可能理解错需求）
- ❌ 只做技术测试，不做场景测试
- ❌ 自己猜测用户需求

**正确做法：**
- ✅ 完整走一遍业务流程
- ✅ 发现问题先讨论
- ✅ 确认方案后再修改
- ✅ 站在最终用户（健身教练）的角度思考

## 🎯 原则 5：数据流向原则 ⭐⭐⭐

**所有页面的数据必须有真实的数据源，不能只做表面工作。**

### 《我的认证》页面数据流
- ✅ 数据来源：《我的训练计划》中的训练记录
- ✅ 计算方式：实时统计 `training_sessions` 表中 `completed=true` 的记录
- ❌ 禁止：手动维护缓存数据，或只显示数字而没有真实数据支撑

**示例：**
- 打卡认证（10次、25次、50次等）→ 统计已完成的训练课次数量
- 体能认证（深蹲、卧推等）→ 统计训练动作中的最大重量/次数
- 影响力认证（推荐会员等）→ 统计实际推荐的会员数量

### 《我的进步》页面数据流
- ✅ 数据来源：《我的训练计划》中的训练记录
- ✅ 显示方式：展示训练动作的历史数据、进步曲线

**工作流程：**
1. 当添加新的认证类型时，必须先确定数据来源
2. 实现数据统计逻辑（从训练计划表中计算）
3. 前端实时查询并显示
4. 测试：添加训练记录 → 认证进度自动更新

### 实时计算 vs 缓存数据

**优先使用实时计算，避免缓存不同步问题。**

#### 实时计算（推荐）✅
- 每次显示时从数据库统计
- 数据永远准确
- 逻辑简单，容易维护
- 适用场景：单个会员的数据量不大（几百条记录）

#### 缓存数据（谨慎使用）⚠️
- 需要触发器或定时任务更新缓存
- 容易出现数据不同步
- 调试困难
- 适用场景：数据量巨大，实时计算性能不足

**当前系统：所有认证进度使用实时计算。**

### 常见错误示例

#### ❌ 错误做法1：只维护缓存表
```javascript
// 手动更新 member_achievement_progress 表
await supabase.from('member_achievement_progress').update({
  current_value: 75,
  is_completed: true
})
```
**问题：** 训练记录删除后，缓存不会自动更新

#### ✅ 正确做法1：实时计算
```javascript
// 实时统计训练记录
const { data: sessions } = await supabase
  .from('training_sessions')
  .select('id')
  .eq('completed', true)

const checkInCount = sessions?.length || 0
```

#### ❌ 错误做法2：只做表面显示
```javascript
// 直接显示硬编码的数字
<div>打卡次数：75</div>
```
**问题：** 没有真实数据支撑

#### ✅ 正确做法2：从数据源计算
```javascript
// 从训练计划统计
const checkInCount = await calculateCheckInCount(memberId)
<div>打卡次数：{{ checkInCount }}</div>
```
